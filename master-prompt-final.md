## Повний контекст для нової розмови

## Хто я

Я founder і одночасно target user цього продукту. Пацієнтка дерматологічної клініки з діагнозом себорейний та акнеформний дерматит. З цього особистого досвіду народилась ідея продукту. Я також розробник.

---

## Проблема

### Одним реченням
Клініка існує для пацієнта до і під час візиту. Після — пацієнт сам по собі.

### Детально
Лікар дає план лікування у вигляді аркуша тексту. 7–10 позицій одночасно: суплементи, засоби для різних зон, шампунь двічі на тиждень замість вмивання, маска тричі на тиждень, LED-протокол раз на 7–14 днів, аналізи в конкретний день циклу.

Пацієнт виходить з клініки без трекера, нагадувань, календаря. Перші дні виконує. Потім — звичайне життя. Пропускає кроки не з байдужості, а з перевантаження. Забуває що минуло три місяці. Записується в іншу клініку — не зі злості, просто вона опинилась зручніше.

**Ключова теза:** Пацієнт не зобов'язаний пам'ятати про клініку. Клініка зобов'язана пам'ятати про пацієнта.

**Для бізнесу:** якщо пацієнт випадає не через незадоволення, а через перевантаження — це втрата повторних візитів, зниження adherence і пряма втрата доходу клініки.

## Що я будую

### Назва
**Care Assistant** — Telegram-first care layer для дерматологічних і косметологічних клінік.
**Proton** — адмін-портал для клініки.

### Позиціонування
Не нова CRM. Не мобільний застосунок. Шар поверх того що клініка вже має. Супроводжує пацієнта від моменту виходу з кабінету до наступного візиту.

### Формула продукту
```
Лікар написав план
    → система структурує
    → календар планує
    → бот супроводжує
    → адмін втручається тільки якщо потрібно
```

### Цільовий ринок
Невеликі дерматологічні та косметологічні клініки. Україна та схожі ринки (Польща, Румунія, Балтика). 2–15 лікарів, 1–2 адміністратори, власник часто є головним лікарем.

## Telegram Bot — повна функціональність

### PATIENT SIDE

**1. Онбординг**
- Клініка надсилає посилання після прийому
- Пацієнт запускає бота, прив'язується до свого запису
- При онбординзі підписує consent: згода на нагадування, дзвінки від бота, зберігання даних
- Якщо жінка: опціонально вводить дані циклу (з окремою згодою) для нагадувань про аналізи

**2. Структурований план лікування**
- Бот надсилає план не як суцільний текст, а структуровано:
  - Ранковий routine (список по порядку)
  - Вечірній routine
  - Дії в конкретні дні тижня (маска — пн/ср/пт)
  - Суплементи з часом прийому
  - Спеціальні дії прив'язані до дат або умов (аналізи на 2-й день циклу)
- Лінк: "Переглянути повний план →" — відкриває web-сторінку з treatment plan
  - Ідентифікація через Telegram або токен у посиланні
  - Пацієнт бачить: виконано / залишилось / скільки часу в клініці / наступні кроки

**3. ICS-файл — календар**
- Генерується автоматично після підтвердження плану лікарем
- Recurring events на весь курс лікування
- Apple Calendar / Google Calendar / Outlook
- Окремо: ICS для кожного запису на процедуру (за тиждень / 2 дні / день / годину до візиту з локацією)

**4. Щоденні нагадування**
- В потрібний час: "Ранковий догляд — не забудь SPF ✅"
- Разові дії: "Сьогодні день маски"
- Суплементи: "Час прийняти Optibac + Glutagenics"
- Спеціальні: "Сьогодні 2-й день циклу — здайте гормональну панель"
- Пацієнт відмічає виконання або пропускає
- **Два режими:**
  - Повний режим — нагадування про процедури + запис + домашній догляд
  - Home care режим — тільки щоденний догляд (якщо пацієнт за кордоном або на паузі)

**5. Treatment tracker**
- Прогрес курсу: "День 14 з 30"
- Відсоток виконання протоколу за тиждень/місяць
- Daily adherence: пацієнт відмічає щодня навіть в home care режимі

**6. Booking flow**
- Підтвердити / перенести / скасувати запис
- Запит до адміністратора якщо потрібне ручне втручання
- Проактивний follow-up від бота: "Минуло 3 місяці. Рекомендований повторний прийом. Записати вас?"

**7. Confirmation flow з ескалацією**
- Рівень 1: Telegram повідомлення з кнопками Так / Ні / Перенести
- Рівень 2: якщо не прочитано за X годин → автоматичний голосовий дзвінок (бот одразу представляється що він бот, пропонує підключити живу людину — "натисніть 2")
- Рівень 3: якщо не відповів → задача адміну в Action Inbox

**8. Нагадування про засоби**
- Бот знає що призначено і тривалість курсу
- Розраховує коли засіб може закінчуватись
- Надсилає нагадування пацієнту і задачу адміну: "Підготуйте список засобів для цього пацієнта"

**9. Win-back для неактивних**
- Пацієнт давно не відповідає → бот надсилає повідомлення
- Є treatment plan → "Згідно вашого плану залишились процедури. Продовжимо?"
- Немає плану → "Ви давно не були. Запрошуємо зі знижкою 10%"
- Відповідь "за кордоном" → автоматична пропозиція паузи + підписка на новини

**10. Document storage**
- Пацієнт може завантажити аналізи і апдейти стану
- Зберігається в захищеній папці прив'язаній до картки
- Лікар бачить перед прийомом

## Patient Card — структура даних

```
Patient Card
    ├── Персональні дані (без діагнозу — щоб уникнути МІС-регуляції)
    ├── Treatment Plan (живий документ)
    │       ├── Pre-care phase
    │       ├── Active treatment phase
    │       └── After-care phase
    ├── Процедури виконані (з датами)
    ├── Відхилення від плану (з позначкою: заплановано чи ні)
    ├── Adherence статистика
    ├── Смета (заплановано / виконано / залишилось)
    └── Документи пацієнта (аналізи, фото)
```

**Treatment Plan як живий документ:**
- Парситься з тексту або фото при введенні
- Редагується лікарем коли змінюється стан
- При кожній зміні автоматично оновлює нагадування боту

**Нотифікації лікарю:**
- "Пацієнт виконав всі процедури — перегляньте план"
- "Виявлено відхилення від плану — підтвердіть"
- "План закінчується через X днів — потрібне оновлення"

## Rules Engine

Окремий компонент який визначає автоматично:
- Чи потрібен допуск лікаря для конкретної процедури
- Які процедури можна поєднувати
- Які обов'язкові інтервали між процедурами
- Які умови для запису (наприклад: не раніше ніж 30 днів після лазеру)
- Яка фаза лікування зараз (pre / active / after-care)

Rules Engine — це те чого немає в жодній існуючій CRM. Клініка не може прописати свою логіку в Beauty Pro або Clinic iQ. Тут — може.

## Proton — адмін-портал

### Концепція
Один портал замість трьох месенджерів. Unified Inbox: Telegram, Instagram DM, Viber — все в одному місці. Адмін відповідає звідти, повідомлення іде у відповідний канал пацієнта.

### UNIFIED INBOX
- Всі повідомлення з усіх каналів
- Статус: автоматично оброблено / потребує відповіді / закрито
- Коли адмін підключається — автоматично засікається час
- Метрика: скільки часу витрачено на ручну комунікацію vs автоматичну

### ACTION INBOX (канбан)
Пріоритети на сьогодні:
- Підтвердити записи на завтра
- Пацієнти що не відповіли на нагадування
- Win-back список (давно не були)
- Нові запити з бота
- Approval запити від лікаря

### APPROVAL FLOW З ЛІКАРЕМ
```
Пацієнт хоче процедуру
    → бот не знає чи є допуск
    → адмін: "уточню у лікаря" + одна команда
    → лікар отримує запит
    → підтверджує/відхиляє
    → в чаті пацієнта: "Лікар підтвердив. Оберіть зручний час"
    → слоти для запису
```

### КАРТКА ПАЦІЄНТА
- Ім'я, контакти, дата народження
- Активний treatment plan
- Статус adherence (відсоток за тиждень)
- Історія візитів і процедур
- Наступний рекомендований візит
- Документи (аналізи, фото)
- Нотатки лікаря

**Додавання плану:**
- Лікар або адмін вводить текст або завантажує фото/PDF
- Система парсить і структурує
- Превью перед відправкою
- Після підтвердження → іде пацієнту + генерується ICS

### РОЗКЛАД
- Синхронізація з CRM клініки через API/webhook
- Графік відвідувань
- Конфлікти в розкладі
- Статус записів: підтверджено / не підтверджено / перенесено

### АНАЛІТИКА (для власника)
- Кількість активних vs неактивних пацієнтів
- Відсоток no-show за місяць
- Середній adherence по пацієнтах
- Час адміна: ручна робота vs автоматика
- Win-back воронка: скільки повернулось / відреагувало на знижку / не відповіло
- Upsell через бота: скільки процедур записано через нагадування

## Treatment Plan — технічний flow генерації

### Як план потрапляє від лікаря до пацієнта

```
Лікар пише план на прийомі (від руки або набирає на пристрої)
    ↓
Завантажується в S3 (тимчасово — тільки для обробки)
    ↓
S3: OCR — розпізнавання тексту з фото або PDF
S3: витягування структурованого тексту
    ↓
LLM промпт: структурує текст у план лікування
    ↓
Генерується два виходи паралельно:
    ├── PDF для пацієнта (таблиці, розклад, процедури, ціни)
    └── Structured JSON для бота (нагадування + Rules Engine)
    ↓
S3 видаляє оригінал після обробки
    ↓
Proton Drive: зберігається PDF + structured data
    ↓
Лікар отримує превью → підтверджує
    ↓
Пацієнт отримує в Telegram:
    ├── Повідомлення зі структурованим планом
    ├── Посилання на PDF (TTL: X годин/днів)
    ├── ICS файл для календаря
    └── Активовані нагадування згідно плану
```

### Що містить PDF для пацієнта

**Блок 1 — Домашній догляд**
Таблиця по часу і днях:

| Час | Дія | Засіб | Примітка |
|---|---|---|---|
| Ранок 07:00 | Вмивання | Lait SR BR | 2-3 хв, двічі підряд |
| Ранок 07:10 | Лосьйон | P50V BR | Перші 3 тижні на мокрий спонж |
| Вечір 21:00 | Маска | Vivant BR | Пн/Ср/Пт — T-зона з содою |

**Блок 2 — Суплементи**
Таблиця: що / коли / доза / тривалість курсу

**Блок 3 — Процедури в клініці**
Перелік: назва / рекомендована дата / ціна / загальна сума

**Блок 4 — Додати в календар**
Посилання або QR → ICS файл на весь курс

### Fallback якщо Telegram недоступний
- PDF надсилається також на email при онбординзі
- Web-лінк з планом доступний незалежно від Telegram
- ICS вже в календарі пацієнта — нагадування працюють офлайн

## Архітектура зберігання даних

### Принцип: медичні дані не проходять через соцмережі

Telegram, Viber, Instagram отримують **тільки:**
- Текстові нагадування
- Посилання з обмеженим TTL
- Кнопки підтвердження

Самі дані (PDF, фото, аналізи) — **тільки Proton Drive.**

### Proton Drive — чому саме він

- End-to-end encryption за замовчуванням
- GDPR-compliant
- Audit log: хто і коли відкривав файл
- Розмежування ролей на рівні папок
- Посилання з TTL: доступ до файлу активний X годин/днів, потім закривається автоматично

### Роль-based доступ

```
Власник клініки
    → аналітика по всіх пацієнтах
    → фінансові звіти
    → доступ до всіх даних

Лікар
    → тільки свої пацієнти
    → медичні дані і фото
    → нотатки і treatment plan

Адміністратор
    → записи і розклад
    → ім'я пацієнта і час запису
    → БЕЗ медичних даних
    → БЕЗ повного ПІБ і контактів (тільки для підтвердження запису)
    → не може експортувати або скачати базу

Пацієнт
    → тільки своя папка
    → свій treatment plan
    → свої аналізи і фото
```

### Аналізи від пацієнта — flow

```
Пацієнт завантажує аналізи в свою папку Proton Drive
    ↓
Лікарю приходить сповіщення: "Пацієнт X додав аналізи"
    ↓
Лікар переглядає → натискає "Переглянуто" → залишає коментар
    ↓
Пацієнт бачить статус: лікар відреагував
```

### Фото прогресу лікування
- Пацієнт або лікар завантажує фото по фазах лікування
- Зберігається в Proton Drive прив'язане до картки і дати
- Лікар бачить динаміку: до / після / в процесі
- Пацієнт бачить свій прогрес (мотивація дотримуватись плану)

## ОКРЕМО: Security Policy для клініки
### (не є частиною Care Assistant — окремий продукт або onboarding послуга)

Це операційна безпека клініки як організації. Може бути оформлена як:
- Окремий консалтинговий документ
- Частина onboarding пакету при підключенні до Care Assistant
- Окремий SaaS продукт для медичних організацій

**Що входить:**
- MDM (Mobile Device Management) — захист пристроїв клініки
- Microsoft / Google Workspace акаунти з розмежуванням доступу для кожної ролі
- Політика "нічого не зберігати локально на пристроях"
- Proton Mail замість Gmail для внутрішньої комунікації клініки
- Proton Drive для всіх медичних документів організації
- Захист від несанкціонованого доступу і вилучення техніки
- Audit logs по всіх діях з даними пацієнтів
- Заборона на експорт бази даних пацієнтів для адмінів

## ОКРЕМО: Security & IT Audit — пілотна клініка Revive (Львів)
### Контекст для нової розмови — НЕ є частиною Care Assistant продукту

Це повний AS-IS опис поточного стану IT та безпеки пілотної клініки. Використовується для:
- Розробки onboarding checklist при підключенні клініки до Care Assistant
- Розуміння технічних ризиків інтеграції
- Потенційного окремого Security Consulting продукту

---

### AS-IS стан клініки

**CRM / Дані / Vendor lock-in:**
- Клініка повністю залежить від CRM операційно
- Міграція без аудиту: BeautyPro → ClinicX → ClinicCards
- Паралельне використання BeautyPro + ClinicX спричинило втрати даних і фінансові збитки
- ClinicCards — поточна система, але без дерматологічної функціональності
- Немає незалежної бази даних, немає backup стратегії, немає NAS, немає cloud replication під контролем клініки
- Втрата доступу до CRM = повна зупинка бізнесу

**MIS / Юридична відповідність:**
- Юридично клініка зобов'язана використовувати МІС
- "Doctor Eleks" оплачений (один акаунт) тільки для юридичної відповідності
- Реально МІС не використовується — всі процеси в CRM
- CRM ≠ МІС = регуляторний ризик
- Біометричні дані (VISIA, дерматоскоп, трихоскоп) зберігаються локально на ПК поза МІС

**Електронний підпис (КЕП):**
- КЕП власниці зберігався як файл на ноутбуці адміністратора
- Ноутбук без firewall, без EDR, без шифрування диску
- Адміністратори підписували фінансові документи власниці на основі усних вказівок по телефону
- Відсутні: hardware token, MFA, audit trail

**Пристрої (endpoints):**
- Немає інвентаризації пристроїв
- Різні версії Windows, ліцензування не перевірено
- Відсутні: MDM, EDR/AV, автоблокування екрану, контроль USB
- Дані зберігаються локально на пристроях

**Мережа:**
- "Формально сегментована" але реально не захищена
- Пароль внутрішнього Wi-Fi роздається вільно включно відвідувачам
- Guest Wi-Fi не ізольований фізично
- Камери: китайський виробник, встановлені стороннім провайдером, невідомо до якої мережі підключені і куди передають відео
- Медичні прилади (VISIA, трихоскоп) в тій самій мережі що адміни
- Відсутні: firewall rules, IDS/IPS, моніторинг мережі, egress controls

**Комунікаційні канали:**
- Пацієнти пишуть через Telegram, Viber, Gmail
- Gmail без MFA
- Вкладення від пацієнтів (PDF, Word, фото аналізів) відкриваються напряму на робочих ПК
- Авто-завантаження в месенджерах може бути увімкнене
- Немає sandboxing, немає сканування вкладень

**Біометричні / медичні дані:**
- Фото пацієнтів і зображення дерматоскопії зберігаються локально
- Папки називаються з повним ПІБ пацієнтів
- Монітори без privacy фільтрів
- USB доступ відкритий — можна скопіювати папки непомітно
- Відсутні: DLP, access controls, audit logs

**Акаунти та доступи:**
- Microsoft і Gmail акаунти створювались хаотично
- Частина акаунтів прив'язана до колишнього чоловіка власниці
- Один Microsoft акаунт прив'язаний до кількох пристроїв включно пристрій колишнього чоловіка
- Власниця платить підписки прив'язані до цього акаунту
- Відсутні: MFA, RBAC, структуровані Windows акаунти

**Фізична безпека:**
- Контроль доступу залежить від електрики
- Під час блекаутів: ненадійний доступ, були випадки несанкціонованого входу
- Немає аварійних протоколів

**Поведінкові ризики:**
- Адміни обговорювали пацієнтів неналежним чином
- Відсутні: NDA дисципліна, навчання з кібербезпеки, навчання з медичної етики даних

### TO-BE вимоги (що клініка хоче)

- CRM залишається операційним ядром
- МІС тільки для юридичної відповідності
- Файли НЕ зберігаються локально: тимчасовий кеш → одразу upload → видалення локально
- Доступ до файлів через expiring links і ID-based access
- Уникати передачі імен/телефонів пацієнтів через месенджери
- Електронне підписання consent на: обробку даних, біометрію, публікацію в соцмережах
- Hardware MFA (YubiKey) для власника/адмінів/привілейованих акаунтів
- Централізована AI-платформа комунікації: пацієнт → бот → CRM → адмін при потребі
- Файли від пацієнтів через sandbox перед тим як хтось відкриє
- Пацієнт відображається як ID (без телефону/імені в месенджері)

### Що це означає для Care Assistant

При інтеграції Care Assistant з клінікою типу Revive:
- Telegram бот НЕ передає медичні дані через Telegram — тільки посилання з TTL
- Всі файли зберігаються в Proton Drive, не локально
- Пацієнт в системі — це ID, не ПІБ
- Адмін бачить тільки те що потрібно для його ролі
- Consent підписується при онбординзі пацієнта в бот
